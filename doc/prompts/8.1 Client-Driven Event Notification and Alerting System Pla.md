# Client-Driven Event Notification and Alerting System Plan

We will implement a robust event notification and alerting system for the UITS Event Raiders platform. The implementation consists of local client-side caching & deadline alerts, OneSignal Web Push SDK integration, and a GitHub Actions workflow with an automated Node.js dispatch script.

## User Review Required

> [!IMPORTANT]
> The GitHub Actions workflow requires setting two GitHub Repository Secrets:
> 1. `ONESIGNAL_APP_ID`: Your OneSignal App ID.
> 2. `ONESIGNAL_REST_API_KEY`: Your OneSignal REST API Key.
> 
> Also, you must configure your public platform base URL in the workflow or environment variables (we will default to auto-detecting it or using a standard default).

> [!NOTE]
> Since standard browser push notifications require HTTPS and user subscription permission, client-side testing of push notifications should be done in a secure environment. For local testing, we have enabled a premium, responsive glassmorphic UI Toast fallback on the page in case browser notifications are blocked or unsupported.

## Proposed Changes

### Configuration Registry

#### [NEW] [tracker.json](Documents/GitHub/%5BoU1TS%5D/events/tracker.json)
Create `tracker.json` at the root of the workspace to track synchronization state:
- `lastUpdated`: ISO timestamp representing the last time a notification was successfully sent.
- `eventsCount`: Total count of active events inside `raids.json` (currently 9).
- `activeReminders`: Array of objects containing:
  - `raidNum` (number)
  - `regEndDate` (string: YYYY-MM-DD)
  - `reminderSent` (boolean flag)

{
  "lastUpdated": "2026-07-12T00:00:00.000Z",

### Client-Side Engine & Caching

#### [MODIFY] [app.js](Documents/GitHub/%5BoU1TS%5D/events/app.js)
Update `app.js` to implement:
1. **Aggressive Cache Validation**:
   - On DOM load, fetch `tracker.json` using `fetch('tracker.json', { cache: 'no-cache' })`.
   - Check `localStorage` under key `ev_tracker`.
   - If the remote `eventsCount` is greater than the local one (or local is empty), clear local state cache, update `localStorage` with remote metadata, and trigger an asynchronous reload of `raids.json` bypassing caches.
2. **Registration Deadline Alerts**:
   - Check all items in `activeReminders`.
   - Parse `regEndDate` relative to local system clock.
   - If local system clock is exactly 1 day prior to `regEndDate`, check if `ev_alert_sent_${raidNum}` exists in `localStorage`.
   - If not found, trigger a localized notification (system notification and/or premium UI toast banner) and store `ev_alert_sent_${raidNum} = true`.

#### [MODIFY] [style.css](Documents/GitHub/%5BoU1TS%5D/events/style.css)
Append premium glassmorphic toast notification styles to the end of `style.css`:
- Modern animations (slide-in from right/bottom, bounce, progress bar).
- Translucent backdrop blur, neon glowing borders matching the current theme accent, gold highlights, and typography.

### Automation Pipeline & Scripting

#### [NEW] [send-push.js](Documents/GitHub/%5BoU1TS%5D/events/.github/scripts/send-push.js)
Create Node.js script to run in GitHub Actions:
- Reads and parses `raids.json`.
- Isolate the most recent array entry (highest `Raid_Num` or index `0`).
- Validates that OneSignal API credentials exist in environment variables.
- Calls OneSignal REST API using the native Node.js `fetch` client to dispatch a push message:
  - **Audience**: Subscribed Users.
  - **Title**: Dynamic title containing raid theme.
  - **Body**: Custom description containing the venue and registration deadline info.
  - **URL**: Destination page (e.g. `https://ou1ts.github.io/events/#raid-[Raid_Num]`).
- Supports an optional `--deadline-check` flag to read `tracker.json` / `raids.json` and send alerts for raids closing tomorrow, updating `tracker.json`'s `reminderSent` flag and `lastUpdated` timestamp.

#### [NEW] [notify.yml](Documents/GitHub/%5BoU1TS%5D/events/.github/workflows/notify.yml)
Create GHA workflow triggered on:
- Merge/push to `raids.json`.
- Daily cron schedule (`0 6 * * *`) for deadline checks.
- Workflow runs Node.js script, updates `tracker.json`, and commits back to repository.
