# Implementation Plan - Fix OneSignal Web Push & Service Worker Permission Prompting

Fix the issue where the live website fails to prompt for notification permissions and push notifications do not work.

## Root Cause Analysis

1. **OneSignal v16 API Incompatibility (`OneSignal` vs `OneSignalDeferred`)**:
   - `index.html` initializes OneSignal using `window.OneSignal = window.OneSignal || []` and `OneSignal.push(...)`.
   - In OneSignal SDK v16 (`OneSignalSDK.page.js`), the initialization queue uses `window.OneSignalDeferred`. Because `window.OneSignal` was used, `OneSignal.init()` was never executed by the SDK when loaded.
2. **Missing Service Worker Path for Subpath Hosting**:
   - On GitHub Pages (`https://ou1ts.github.io/events/`), OneSignal defaults to looking for `/OneSignalSDKWorker.js` at root domain level (`https://ou1ts.github.io/OneSignalSDKWorker.js`), causing a 404 error.
   - `serviceWorkerPath` and `serviceWorkerParam` must be specified in `OneSignal.init()`.
3. **Implicit Permission Prompting**:
   - OneSignal v16 requires explicit permission prompt configuration (`slidedownOptions` or `OneSignal.Notifications.requestPermission()`) so the browser prompts the user to grant notification permissions.

## Proposed Changes

### Frontend Configuration (`index.html`)

#### [MODIFY] [index.html](file:///c:/Users/.../Documents/GitHub/%5BoU1TS%5D/events/index.html)
- Replace `window.OneSignal` queue with `window.OneSignalDeferred = window.OneSignalDeferred || [];`
- Update `OneSignalDeferred.push(async function(OneSignal) { ... })` for v16 asynchronous SDK compatibility.
- Configure `serviceWorkerPath: "OneSignalSDKWorker.js"` and `serviceWorkerParam: { scope: "./" }` inside `OneSignal.init()`.
- Add `slidedown` prompt configuration and explicit `OneSignal.Notifications.requestPermission()` handling if needed to prompt the user for permission.
