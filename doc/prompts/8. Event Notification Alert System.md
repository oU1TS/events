### 1. Data Structure Setup
- Create a `tracker.json` file to manage synchronization states. It must contain:
  - `lastUpdated`: ISO timestamp updated whenever an alert is successfully dispatched.
  - `eventsCount`: A numerical count reflecting the absolute total of active "Raid_Num" entries inside `events/raids.json`.
  - `activeReminders`: An array of objects tracking upcoming deadlines containing `raidNum`, `regEndDate` (YYYY-MM-DD), and a `reminderSent` boolean flag.

### 2. Client-Side Caching & Local Storage Engine
Implement a high-performance, cache-first local validation layer in vanilla JavaScript:
- Aggressive Caching Strategy: On document load, bypass standard browser asset caches to fetch a completely fresh instance of `tracker.json` (`cache: 'no-cache'`).
- State Validation: Compare the fetched `eventsCount` and `lastUpdated` values against values stored inside the user's `localStorage` (key: `ev_tracker`).
- State Update Protocol: If the remote file contains a higher `eventsCount`, immediately flush the stale local cache data, update `localStorage` with the newest configuration parameters, and trigger a UI refresh asynchronously.
- Registration Deadline Evaluation: Check all elements in `activeReminders`. If the current client-side system clock is exactly 1 day prior to any `regEndDate`, cross-reference `localStorage` to check if a specific deadline alert has already fired. If not, trigger a localized notification warning and commit a token flag to storage to prevent redundant notifications.

### 3. OneSignal Integration & Background Workers
- Integrate Web Push functionality using `OneSignalSDK.page.js`. Provide the initialization script blocks utilizing standard initialization parameters (`appId`, `notifyButton`, etc.).
- Set up the environment infrastructure assuming a pre-built `OneSignalSDKWorker.js` service worker script is loaded globally at the root directory level, enabling true background push message registration across compatible desktop and mobile web browsers.

### 4. GitHub Actions Automated Pipeline
Create a fully decoupled backend dispatch runner using GitHub Actions workflows to monitor repository modifications:
- Workflow File (`.github/workflows/notify.yml`): Configure an execution pipeline triggered automatically whenever changes are successfully merged into `events/raids.json`. Additionally, set a daily cron schedule (`0 6 * * *`) to scan for upcoming registration deadline dates.
- Automation Engine Script (`.github/scripts/send-push.js`): Write a lightweight Node.js utility executed by the runner. The script must:
  - Dynamically read and parse the updated `events/raids.json`.
  - Isolate the single most recent array entry (the newly deployed raid configuration).
  - Use the native `fetch` API to make an authorized POST call to the OneSignal REST API (`https://onesignal.com/api/v1/notifications`).
  - Extract credentials securely via server environment targets: `process.env.ONESIGNAL_APP_ID` and `process.env.ONESIGNAL_REST_API_KEY`.
  - Construct a broadcast payload targeting "Subscribed Users", passing a dynamic title containing the raid theme, a tailored body description containing the venue, and a destination property directing users back to the platform base URL.

Generate clean, well-documented, production-grade files across the following components: 'tracker.json', 'app.js' (notification and caching logic), '.github/scripts/send-push.js', and '.github/workflows/notify.yml'.
